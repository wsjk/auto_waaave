2. 🚀 Application Setup & Initialization

The openFrameworks Lifecycle

📖 Understanding the oF App Lifecycle

Every openFrameworks app follows this pattern:

  1. setup() - Called once when the app starts
  2. update() - Called every frame to update logic
  3. draw() - Called every frame to render graphics
  4. exit() - Called when the app closes

Think of it like a game loop: the system repeatedly calls update() then draw() at your target framerate (30 FPS in this project).

Main Entry Point

// src/main.cpp - The application entry point

#include "ofMain.h"
#include "ofApp.h"

int main() {
    // Step 1: Configure OpenGL ES 2.0 for Raspberry Pi
    // Why ES 2.0? Raspberry Pi uses embedded GPU that supports GLES
    ofGLESWindowSettings settings;
    settings.glesVersion = 2;

    // Step 2: Set output resolution (CEA mode 2 = 720x480)
    // This must match your display's capabilities
    settings.setSize(720, 480);

    // Step 3: Create the window with these settings
    ofCreateWindow(settings);

    // Step 4: Create and run the application
    // This starts the setup() → update() → draw() loop
    ofRunApp(new ofApp());
}
⚠️ Memory Management Note

Notice new ofApp() - we're dynamically allocating memory. openFrameworks will handle cleanup automatically, but in general C++ practice, every new should have a corresponding delete. Here, oF manages this for us.

Application Setup Function

// src/ofApp.cpp - setup() is called once at startup

void ofApp::setup() {
    // STEP 1: Configure frame rate
    // Why 30 FPS? Balance between smoothness and CPU/GPU load on RPi
    ofSetFrameRate(30);

    // STEP 2: Set background color (black = 0)
    ofBackground(0);

    // STEP 3: Hide mouse cursor for cleaner display
    ofHideCursor();

    // STEP 4: Initialize subsystems in order
    // Order matters! Each depends on previous setup
    inputSetup();           // Configure camera/video input
    midiSetup();            // Connect to MIDI controller
    fbDeclareAndAllocate(); // Allocate GPU memory for framebuffers

    // STEP 5: Load GPU shaders from disk
    // These are compiled by the GPU at runtime
    shader_mixer.load("shadersES2/shader_mixer");
    shaderSharpen.load("shadersES2/shaderSharpen");

    // STEP 6: Setup audio analysis
    fft.setup();
    fft.setNormalize(false); // We'll manually normalize later

    // STEP 7: Initialize parameter automation arrays
    p_lockClear();
    midiLatchClear();
}
✅ Initialization Best Practice

Notice the careful ordering of initialization. This pattern prevents common bugs:

  • Initialize simple state first (frame rate, colors)
  • Then hardware interfaces (camera, MIDI)
  • Then GPU resources (framebuffers, shaders)
  • Finally, data structures that depend on everything else

If you initialize framebuffers before setting the resolution, you might allocate the wrong size!

3. 🎞️ Framebuffer Management & Circular Buffers

What is a Framebuffer?

💡 Framebuffer (FBO) Explained

A Framebuffer Object (FBO) is like a "virtual canvas" in GPU memory. Instead of drawing directly to the screen, you can draw to an FBO. This lets you:

  • Render multiple "layers" of graphics
  • Apply effects to rendered content
  • Store frames for later use (our video delay effect)
  • Chain multiple effects together

Think of it like working in Photoshop layers - you can composite, blend, and process each layer independently.

The Circular Buffer Pattern

The core of this application is a circular buffer (also called a ring buffer). This is a classic data structure for implementing video delay effects.

Circular Buffer Visualization (60 frames)

    Current Frame (write position)
           ↓
    [0][1][2][3][4]...[57][58][59]
     ↑                           ↑
     └───────────────────────────┘
     When we reach frame 59, we wrap
     back to frame 0 (circular!)
            
// Global variables at file scope

const int framebufferLength = 60; // Store 60 frames (2 seconds @ 30fps)

// The actual buffer - an array of framebuffer objects
ofFbo pastFrames[framebufferLength];

// Tracking variables for circular indexing
unsigned int framecount = 0;       // Total frames since start
int framedelayoffset = 0;           // Current position in circular buffer

// Function called every frame to advance the buffer
void incIndex() {
    // STEP 1: Increment total frame count
    framecount++;

    // STEP 2: Use modulo to wrap around
    // Example: frame 61 % 60 = frame 1 (we loop back!)
    // This is the "magic" of circular buffers
    framedelayoffset = framecount % framebufferLength;
}
🧠 Understanding Modulo Operator (%)

The modulo operator % gives you the remainder after division:

  • 0 % 60 = 0
  • 59 % 60 = 59
  • 60 % 60 = 0 ← wraps back to start!
  • 61 % 60 = 1
  • 119 % 60 = 59

This creates a repeating cycle of 0-59, perfect for our circular buffer.

Allocating Framebuffers

void ofApp::fbDeclareAndAllocate() {
    // STEP 1: Allocate main processing buffers
    // These are single framebuffers for various stages

    framebuffer0.allocate(width, height);
    framebuffer0.begin();
    ofClear(0, 0, 0, 255); // Clear to black (RGBA)
    framebuffer0.end();

    aspect_fix_fbo.allocate(width, height);
    dry_framebuffer.allocate(width, height);
    sharpenFramebuffer.allocate(width, height);

    // STEP 2: Allocate the circular buffer array
    // Why loop? We need 60 separate framebuffers in GPU memory!
    for (int i = 0; i < framebufferLength; i++) {
        pastFrames[i].allocate(width, height);

        // Initialize each to black
        // begin() = "start drawing to this FBO"
        // end() = "stop drawing to this FBO"
        pastFrames[i].begin();
        ofClear(0, 0, 0, 255);
        pastFrames[i].end();
    }
}
💾 Memory Usage Calculation

Each framebuffer at 640x480 with RGBA (4 bytes per pixel) takes:

640 × 480 × 4 bytes = 1,228,800 bytes ≈ 1.2 MB

With 60 frames: 60 × 1.2 MB = 72 MB of GPU memory

Plus additional buffers (dry, sharpen, etc.) brings total to ~80-90 MB. This is why the Raspberry Pi needs sufficient GPU memory allocation in config.txt!

Reading from the Buffer with Delay

// Calculate which frame to read for a delay effect

int fb0_delayamount = 30; // User-controlled delay (0-59 frames)

// Complex formula explained step-by-step:

// STEP 1: Start with current write position
int current = framedelayoffset;

// STEP 2: Subtract the delay amount to go back in time
// If delay = 30 and current = 40, we want frame 10
int target = current - fb0_delayamount;

// STEP 3: Handle negative wraparound
// If current = 10 and delay = 30, we get -20
// We need to wrap to frame 40 (60 - 20)
d_delay = (abs(framedelayoffset - framebufferLength - fb0_delayamount) - 1) % framebufferLength;

// STEP 4: Use this index to read the delayed frame
shader_mixer.setUniformTexture("fb", pastFrames[d_delay].getTexture(), 1);
🎯 Why This Pattern is Powerful

Circular buffers are incredibly efficient:

  • No memory allocation per frame - we reuse the same 60 slots
  • Constant time access - reading any frame is O(1)
  • No copying - frames stay in GPU memory
  • Perfect for real-time - no garbage collection or delays

This is the same technique used in audio buffers, network packet queues, and video codecs.

4. 🎨 GPU Shaders & Video Effects

What are Shaders?

💡 Introduction to GPU Shaders

Shaders are small programs that run on the GPU (Graphics Processing Unit), not the CPU. They process pixels in parallel - potentially thousands at once!

Key Concepts:

  • Vertex Shader - Processes vertex positions (we don't modify these much)
  • Fragment Shader - Processes each pixel's color (this is where the magic happens)
  • GLSL - OpenGL Shading Language (looks like C)

Think of it like this: Your CPU is a smart executive (good at complex decisions), while your GPU is thousands of simple workers (great at doing the same task on lots of data).

The Vertex Shader (Simple Pass-through)

// shadersES2/shader_mixer.vert
// This runs once per vertex of the rectangle we're drawing

// INPUT: Position and texture coordinates from openFrameworks
attribute vec4 position;      // Where is this vertex?
attribute vec2 texcoord;     // What part of the texture does it map to?

// OUTPUT: Pass texture coords to fragment shader
varying vec2 texCoordVarying;

// Matrix to transform from 3D space to screen space
uniform mat4 modelViewProjectionMatrix;

void main() {
    // Pass texture coordinates to fragment shader
    // "varying" means this value is interpolated across the surface
    texCoordVarying = texcoord;

    // Transform vertex position to screen coordinates
    gl_Position = modelViewProjectionMatrix * position;
}
🔍 Understanding Varying Variables

A varying variable is automatically interpolated across the surface. For a rectangle with corners at (0,0), (1,0), (0,1), (1,1):

  • Top-left corner: texCoordVarying = (0, 0)
  • Top-right corner: texCoordVarying = (1, 0)
  • Center: texCoordVarying = (0.5, 0.5)

The GPU automatically calculates in-between values for every pixel!

The Fragment Shader (Where Magic Happens)

The fragment shader processes EVERY PIXEL. At 640×480, that's 307,200 pixels per frame, 30 times per second = 9.2 million pixels per second!

// shadersES2/shader_mixer.frag (simplified)

precision highp float; // Use high precision (important for color accuracy)

// INPUTS: Textures (video frames)
uniform sampler2D tex0;           // Live camera input
uniform sampler2D fb;             // Delayed feedback frame
uniform sampler2D temporalFilter; // Previous output frame

// INPUTS: Control parameters (from CPU)
uniform float fbMix;              // Blend amount (0-1)
uniform float fbHue;              // Hue shift multiplier
uniform float fbSaturation;       // Saturation multiplier
uniform float fbBright;           // Brightness multiplier
uniform float fbZDisplace;        // Zoom amount
uniform float fbXDisplace;        // Horizontal shift
uniform float fbYDisplace;        // Vertical shift
uniform float fbRotate;           // Rotation angle

// INPUT: Pixel position (from vertex shader)
varying vec2 texCoordVarying;

// MAIN FUNCTION - runs for EVERY pixel!
void main() {
    // STEP 1: Sample the camera input at this pixel
    vec4 input1Color = texture2D(tex0, texCoordVarying);

    // STEP 2: Transform coordinates for feedback frame
    vec2 fbCoord = texCoordVarying;

    // STEP 3: Apply zoom (displacement in Z)
    fbCoord.x = fbZDisplace * (fbCoord.x - 0.5) + 0.5;
    fbCoord.y = fbZDisplace * (fbCoord.y - 0.5) + 0.5;

    // STEP 4: Sample the delayed feedback frame
    vec4 fbColor = texture2D(fb, fbCoord);

    // STEP 5: Blend camera and feedback
    vec4 color = mix(input1Color, fbColor, fbMix);

    // STEP 6: Output final color for this pixel
    gl_FragColor = color;
}
⚡ Why Shaders are Fast

GPU parallelism is incredible. Consider this:

  • CPU approach: Process 307,200 pixels one-by-one = SLOW
  • GPU approach: Process ALL pixels simultaneously = FAST

The Raspberry Pi's GPU can process all pixels in about 1-2 milliseconds, leaving 31ms for everything else in a 30fps frame budget!

Color Space Conversion: RGB ↔ HSB

One of the most important techniques in video processing is converting between color spaces. RGB (Red, Green, Blue) is how computers store colors, but HSB (Hue, Saturation, Brightness) is more intuitive for artistic control. 

// Convert RGB to HSB (Hue, Saturation, Brightness)
// WHY? HSB lets us modify colors more intuitively:
// - Hue: the actual color (red, blue, green, etc.)
// - Saturation: how vivid (0 = grayscale, 1 = full color)
// - Brightness: how light (0 = black, 1 = full brightness)

vec3 rgb2hsb(in vec3 c) {
    // Step 1: Find which RGB channel has max value
    // This determines the hue sector (red, green, or blue zone)
    vec4 K = vec4(0.0, -1.0 / 3.0, 2.0 / 3.0, -1.0);
    vec4 p = mix(vec4(c.bg, K.wz), vec4(c.gb, K.xy), step(c.b, c.g));
    vec4 q = mix(vec4(p.xyw, c.r), vec4(c.r, p.yzx), step(p.x, c.r));

    // Step 2: Calculate hue from the differences between channels
    float d = q.x - min(q.w, q.y);
    float e = 1.0e-10; // Tiny value to prevent division by zero

    // Step 3: Return HSB values
    return vec3(
        abs(q.z + (q.w - q.y) / (6.0 * d + e)), // Hue (0-1)
        d / (q.x + e),                             // Saturation (0-1)
        q.x                                        // Brightness (0-1)
    );
}

// Convert back from HSB to RGB
vec3 hsb2rgb(in vec3 c) {
    // Create rainbow gradient using fractional hue values
    vec4 K = vec4(1.0, 2.0 / 3.0, 1.0 / 3.0, 3.0);
    vec3 p = abs(fract(c.xxx + K.xyz) * 6.0 - K.www);

    // Mix between white and hue based on saturation
    // Then multiply by brightness
    return c.z * mix(K.xxx, clamp(p - K.xxx, 0.0, 1.0), c.y);
}
🎨 Why HSB for Effects?

Example: Making a color more vibrant

  • RGB approach: Increase each channel? But by how much? R+50, G+30, B-20? It's confusing!
  • HSB approach: Just multiply saturation by 1.5. Simple!

Example: Shifting all colors

  • RGB approach: Complex matrix multiplication
  • HSB approach: Just add to hue value (hue = hue + 0.1). Red becomes orange!

The Complete Shader Pipeline

void main() {
    /********** STEP 1: SAMPLE INPUT **********/
    vec4 input1Color = texture2D(tex0, texCoordVarying);
    vec3 input1ColorHsb = rgb2hsb(input1Color.rgb);

    // Store brightness for video-reactive effects
    // VVV = how bright this pixel is (0-1)
    float VVV = input1ColorHsb.z;

    /********** STEP 2: COORDINATE TRANSFORMATION **********/
    // Start with current pixel position (0-1 range)
    vec2 fbCoord = vec2(texCoordVarying.x - 0.5, texCoordVarying.y - 0.5);

    // Apply zoom (larger fbZDisplace = zoom in)
    fbCoord.x = fbZDisplace * fbCoord.x * (1.0 + vZ * VVV);
    fbCoord.y = fbZDisplace * fbCoord.y * (1.0 + vZ * VVV);

    // Apply horizontal/vertical mirror effects
    if (horizontalMirror == 1) {
        if (fbCoord.x > 0.0) {
            fbCoord.x = -1.0 * fbCoord.x; // Flip right side to left
        }
    }

    // Apply displacement (pan effect)
    fbCoord = vec2(
        fbCoord.x + fbXDisplace + (vX * VVV) + 0.5,
        fbCoord.y + fbYDisplace + (vY * VVV) + 0.5
    );

    // Apply rotation
    fbCoord = rotate(fbCoord, fbRotate + (vRotate * VVV));

    /********** STEP 3: HANDLE COORDINATE WRAPPING **********/
    if (toroidSwitch == 1) {
        // Toroidal wrap: coordinates wrap around like a donut
        // If you go off the right edge, you appear on the left
        fbCoord = fract(fbCoord); // Keep only fractional part (0-1)
    }

    /********** STEP 4: SAMPLE FEEDBACK FRAME **********/
    vec4 fbColor = texture2D(fb, fbCoord);

    // Clamp out-of-bounds to black (prevents edge artifacts)
    if (toroidSwitch == 0 && mirrorSwitch == 0) {
        if (fbCoord.x > 1.0 || fbCoord.y > 1.0 ||
            fbCoord.x < 0.0 || fbCoord.y < 0.0) {
            fbColor = vec4(0.0);
        }
    }

    /********** STEP 5: COLOR MANIPULATION IN HSB SPACE **********/
    vec3 fbColorHsb = rgb2hsb(fbColor.rgb);

    // Modify hue (color rotation + chaotic oscillation)
    fbColorHsb.x = abs(fbColorHsb.x * fbHue * (1.0 + vHue * VVV) +
                      (fbHuexLfo + vHuexLfo * VVV) * sin(fbColorHsb.x / 3.14));
    fbColorHsb.x = fract(mod(fbColorHsb.x, fbHuexMod + vHuexMod * VVV) +
                        fbHuexOff + vHuexOff * VVV);

    // Modify saturation and brightness
    fbColorHsb.y = clamp(fbColorHsb.y * fbSaturation * (1.0 + vSat * VVV), 0.0, 1.0);
    fbColorHsb.z = clamp(fbColorHsb.z * fbBright * (1.0 + vBright * VVV), 0.0, 1.0);

    // Apply invert switches
    if (brightInvert == 1) { fbColorHsb.z = 1.0 - fbColorHsb.z; }
    if (saturationInvert == 1) { fbColorHsb.y = 1.0 - fbColorHsb.y; }
    if (hueInvert == 1) { fbColorHsb.x = fract(abs(1.0 - fbColorHsb.x)); }

    // Convert back to RGB
    fbColor = vec4(hsb2rgb(fbColorHsb), 1.0);

    /********** STEP 6: TEMPORAL FILTERING (FEEDBACK) **********/
    vec4 temporalFilterColor = texture2D(temporalFilter, texCoordVarying);

    // Boost the resonance (makes feedback more intense)
    vec3 temporalFilterColorHsb = rgb2hsb(temporalFilterColor.rgb);
    temporalFilterColorHsb.z = clamp(
        temporalFilterColorHsb.z * (1.0 + 0.5 * temporalFilterResonance *
        (1.0 + vFb1X * VVV)), 0.0, 1.0);
    temporalFilterColor = vec4(hsb2rgb(temporalFilterColorHsb), 1.0);

    /********** STEP 7: FINAL MIXING **********/
    // Mix camera input with processed feedback
    vec4 color = mix(input1Color, fbColor, fbMix + (vMix * VVV));

    // Luminance keying (like green screen, but based on brightness)
    if (lumakeyInvertSwitch == 0) {
        if (VVV < lumakey + (vLumakey * VVV)) {
            color = fbColor; // Dark pixels show feedback only
        }
    }

    // Add temporal filter into final mix
    color = mix(color, temporalFilterColor,
                temporalFilterMix + (vtemporalFilterMix * VVV));

    // OUTPUT: Set this pixel's color
    gl_FragColor = color;
}
🎯 Key Shader Concepts Recap
  1. Coordinate Transformation: We manipulate WHERE we sample from (zoom, rotate, displace)
  2. Color Manipulation: We manipulate WHAT colors we see (HSB adjustments)
  3. Mixing: We blend multiple sources (camera + feedback + temporal filter)
  4. Video Reactivity: Using input brightness (VVV) to modulate all effects

The Sharpen Shader

The sharpen shader uses a convolution kernel to enhance edges.

// shadersES2/shaderSharpen.frag

void main() {
    /********** STEP 1: SAMPLE NEIGHBORING PIXELS **********/
    // Sample 4 diagonal neighbors (corners of a square)
    float X = 2.0 * 0.0015625; // Offset in X direction
    float Y = 2.0 * 0.0020833; // Offset in Y direction

    // Get brightness of 4 neighbors and average them
    float colorSharpenBright =
        rgb2hsb(texture2D(tex0, texCoordVarying + vec2(X, Y)).rgb).z +
        rgb2hsb(texture2D(tex0, texCoordVarying + vec2(-X, Y)).rgb).z +
        rgb2hsb(texture2D(tex0, texCoordVarying + vec2(-X, -Y)).rgb).z +
        rgb2hsb(texture2D(tex0, texCoordVarying + vec2(X, -Y)).rgb).z;

    colorSharpenBright = colorSharpenBright * 0.125; // Average (4 samples)

    /********** STEP 2: SHARPEN LOGIC **********/
    // Sharpening algorithm:
    // - If center pixel is brighter than neighbors: make it brighter
    // - If center pixel is darker than neighbors: make it darker
    // This enhances edges!

    vec4 ogColor = texture2D(tex0, texCoordVarying);
    vec3 ogColorHSB = rgb2hsb(ogColor.rgb);
    float VVV = ogColorHSB.z;

    // Subtract neighbor average (creates edge detection)
    ogColorHSB.z -= sharpenAmount * colorSharpenBright +
                    (colorSharpenBright * vSharpenAmount * VVV);

    // Boost overall brightness to compensate
    if (sharpenAmount > 0.0) {
        ogColorHSB.z *= (1.0 + sharpenAmount * 0.45 + 0.45 * (vSharpenAmount * VVV));
        ogColorHSB.x *= (1.0 + sharpenAmount * 0.25 + 0.25 * (vSharpenAmount * VVV));
    }

    // Convert back and output
    gl_FragColor = vec4(hsb2rgb(ogColorHSB), 1.0);
}
🔍 How Sharpening Works

Sharpening is based on high-pass filtering:

  1. Sample the pixel's neighbors
  2. Calculate the average
  3. Subtract the average from the center pixel
  4. This removes low-frequency (smooth) information, leaving only edges
  5. Add this edge information back to the original

Result: Edges are enhanced, smooth areas remain smooth.

5. 🎹 MIDI Control System

What is MIDI?

💡 MIDI Explained

MIDI (Musical Instrument Digital Interface) is a protocol for sending control messages. Originally designed for synthesizers, it's perfect for live control of visual effects.

Key concepts:

  • Control Change (CC): A knob or slider sends values 0-127
  • Note On/Off: Keys send note presses (we don't use these here)
  • Channels: 16 separate channels for different controllers

Think of MIDI like a remote control: pressing buttons sends numbers to the application.

MIDI Setup

// Initialize MIDI system
void ofApp::midiSetup() {
    // STEP 1: List all connected MIDI devices (for debugging)
    midiIn.listInPorts();

    // STEP 2: Open MIDI port
    // Port 1 is usually the first MIDI controller
    // You may need to change this number for your setup
    midiIn.openPort(1);

    // STEP 3: Configure what messages we want to receive
    // ignoreTypes(sysex, timing, activeSense)
    // We want ALL messages, so set all to false
    midiIn.ignoreTypes(false, false, false);

    // STEP 4: Register this class as a listener
    // When MIDI messages arrive, newMidiMessage() will be called
    midiIn.addListener(this);

    // STEP 5: Print messages to console (helpful for debugging)
    midiIn.setVerbose(true);
}

MIDI Message Processing

// This function is called automatically when a MIDI message arrives
// It's a CALLBACK - we don't call it, the MIDI system does

void ofApp::newMidiMessage(ofxMidiMessage& msg) {
    // Add message to queue
    midiMessages.push_back(msg);

    // Keep only recent messages (prevent memory overflow)
    while (midiMessages.size() > 2) {
        midiMessages.erase(midiMessages.begin());
    }
}
⚠️ Callback Functions

A callback is a function that YOU write but SOMEONE ELSE calls. In this case, the ofxMidi library calls newMidiMessage() when data arrives.

This is an event-driven pattern. Instead of constantly checking "Is there MIDI data?", we say "Call this function WHEN there's MIDI data."

Common in: GUI systems (onClick), network code (onDataReceived), audio (onAudioBuffer).

MIDI Control Mapping

The real work happens in midibiz(), called every frame to process MIDI messages:

// Process all pending MIDI messages
void ofApp::midibiz() {
    // Loop through all messages received since last frame
    for (unsigned int i = 0; i < midiMessages.size(); ++i) {
        ofxMidiMessage& message = midiMessages[i];

        // We only care about Control Change messages
        if (message.status == MIDI_CONTROL_CHANGE) {

            /********** EXAMPLE: Video Reactive Switch **********/
            if (message.control == 39) { // CC#39 on controller
                if (message.value == 127) {
                    videoReactiveSwitch = 1;    // Button pressed
                    p_lock_0_switch = 0;         // Disable parameter locks
                }
                if (message.value == 0) {
                    videoReactiveSwitch = 0;    // Button released
                    p_lock_0_switch = 1;         // Enable parameter locks
                }
            }

            /********** EXAMPLE: Continuous Control with Latching **********/
            // This implements "soft takeover" - prevents parameter jumping
            if (message.control == 16) { // CC#16 controls lumakey

                // Convert MIDI value (0-127) to normalized float (0.0-1.0)
                float normalizedValue = message.value / 127.0f;

                // SOFT TAKEOVER LOGIC:
                // Only respond when knob value is close to current value
                // This prevents sudden jumps when switching between controls

                if (audioReactiveControlSwitch == 0) {
                    // Normal MIDI control mode

                    if (p_lock_0_switch == 1) {
                        // Check if knob is close to current value
                        float currentValue = p_lock[0][p_lock_increment];
                        float difference = abs(normalizedValue - currentValue);

                        if (difference < CONTROL_THRESHOLD) {
                            // We're close! Enable control (latch on)
                            midiActiveFloat[0] = 1;
                        }

                        // Only update if latched
                        if (midiActiveFloat[0] == 1) {
                            p_lock[0][p_lock_increment] = normalizedValue;
                        }
                    }
                }
            }
        }
    }
}
🎯 Soft Takeover Explained

Imagine you have a knob on your MIDI controller that controls brightness:

  1. Knob is at 50%, brightness is at 50%
  2. You switch to control a different parameter (saturation)
  3. You turn the knob to 80% for saturation
  4. You switch back to brightness control
  5. Without soft takeover: Brightness jumps from 50% to 80% instantly!
  6. With soft takeover: Nothing happens until you move the knob to ~50%, then it "latches" and responds smoothly

This prevents jarring jumps in live performance!

MIDI Value Transformation

// MIDI values are 0-127 (7-bit integers)
// We need to convert these to useful ranges

// PATTERN 1: Unipolar (0.0 to 1.0)
float unipolar = message.value / 127.0f;

// PATTERN 2: Bipolar (-1.0 to 1.0)
// Uses "MIDI_MAGIC" constant (63.5 = middle of 0-127)
#define MIDI_MAGIC 63.50f
float bipolar = (message.value - MIDI_MAGIC) / MIDI_MAGIC;

// Examples:
// value = 0   → bipolar = -1.0 (far left)
// value = 63  → bipolar = -0.01 (almost center)
// value = 64  → bipolar = 0.01 (almost center)
// value = 127 → bipolar = 1.0 (far right)

6. 🔊 Audio Reactivity & FFT

What is FFT?

💡 Fast Fourier Transform (FFT) Explained

FFT converts audio from the time domain (amplitude over time) to the frequency domain (which frequencies are present).

Simple analogy:

  • Time domain: "The audio is loud at 0.5 seconds"
  • Frequency domain: "There's a lot of bass (100Hz) and some treble (8kHz)"

We use this to make visuals react to bass, mids, and treble separately!

FFT Setup and Processing

// In setup()
fft.setup();
fft.setNormalize(false); // We'll manually normalize

// In update() - called every frame
fft.update(); // Analyze the latest audio buffer

// Get frequency bands
void ofApp::fftAssignValues() {
    /********** STEP 1: GET RAW VALUES **********/
    fftLow = fft.getLowVal();    // Bass frequencies (20-250 Hz)
    fftMid = fft.getMidVal();    // Mid frequencies (250-4000 Hz)
    fftHigh = fft.getHighVal();  // High frequencies (4000-20000 Hz)

    /********** STEP 2: CLAMP TO RANGE **********/
    // Limit to 1.0 - range to prevent crazy values
    // "range" is typically 200, so we're working with 1-200
    fftLow = ofClamp(fftLow, 1.0f, range);
    fftMid = ofClamp(fftMid, 1.0f, range);
    fftHigh = ofClamp(fftHigh, 1.0f, range);

    /********** STEP 3: NORMALIZE TO 0-1 **********/
    fftLow = fftLow / range;
    fftMid = fftMid / range;
    fftHigh = fftHigh / range;

    /********** STEP 4: APPLY SMOOTHING **********/
    // Without smoothing, values jump around too much
    // This creates a "moving average" effect

    // Formula: new = (current * rate) + (previous * (1 - rate))
    // smoothing_rate = 0.8 means 80% new, 20% old
    low_value_smoothed = fftLow * smoothing_rate +
                         (1.0f - smoothing_rate) * low_value_smoothed;

    mid_value_smoothed = fftMid * smoothing_rate +
                         (1.0f - smoothing_rate) * mid_value_smoothed;

    high_value_smoothed = fftHigh * smoothing_rate +
                          (1.0f - smoothing_rate) * high_value_smoothed;
}
🧮 Smoothing Math Breakdown

Let's trace through an example with smoothing_rate = 0.8:

  1. Frame 1: fftLow = 0.5, low_value_smoothed = 0 (start)
    new = (0.5 × 0.8) + (0 × 0.2) = 0.4
  2. Frame 2: fftLow = 0.8, low_value_smoothed = 0.4
    new = (0.8 × 0.8) + (0.4 × 0.2) = 0.72
  3. Frame 3: fftLow = 0.9, low_value_smoothed = 0.72
    new = (0.9 × 0.8) + (0.72 × 0.2) = 0.864

The value gradually "catches up" to the target, creating smooth transitions.

7. 🔒 Parameter Locks (Automation)

What are Parameter Locks?

💡 Parameter Lock Concept

Parameter Locks (P-Locks) are a way to record and playback automation. Inspired by Elektron music hardware, they let you:

  1. Start recording
  2. Move your MIDI controllers
  3. Stop recording
  4. The movement loops automatically!

This creates evolving, generative visuals without constant manual control.

P-Lock Data Structure

// P-Lock configuration
const int p_lock_number = 17;   // 17 parameters can be automated
const int p_lock_size = 240;     // 240 steps per loop (8 seconds @ 30fps)

// 2D array: [which_parameter][time_step]
float p_lock[p_lock_number][p_lock_size];

// Smoothed output values (what we actually use)
float p_lock_smoothed[p_lock_number];

// Current position in the loop
int p_lock_increment = 0;

// Smoothing amount (0 = no smoothing, 1 = max smoothing)
float p_lock_smooth = 0.5;

// Is recording enabled?
bool p_lock_switch = 0;

P-Lock Initialization

// Clear all P-Lock data (called at startup)
void ofApp::p_lockClear() {
    // STEP 1: Zero out the entire 2D array
    for (int i = 0; i < p_lock_number; i++) {       // For each parameter
        for (int j = 0; j < p_lock_size; j++) {     // For each time step
            p_lock[i][j] = 0;
        }
    }

    // STEP 2: Zero smoothed outputs
    for (int i = 0; i < p_lock_number; i++) {
        p_lock_smoothed[i] = 0;
    }
}

P-Lock Recording and Playback

// Called every frame to update P-Lock playback
void ofApp::p_lockUpdate() {
    /********** STEP 1: READ AND SMOOTH ALL PARAMETERS **********/
    for (int i = 0; i < p_lock_number; i++) {
        // Read the value at current time step
        float raw_value = p_lock[i][p_lock_increment];

        // Apply exponential smoothing
        // This prevents stair-stepping between recorded values
        p_lock_smoothed[i] = raw_value * (1.0f - p_lock_smooth) +
                             p_lock_smoothed[i] * p_lock_smooth;

        // Kill tiny values to prevent numerical drift
        if (abs(p_lock_smoothed[i]) < 0.05) {
            p_lock_smoothed[i] = 0;
        }
    }
}

// In draw() - advance the playback position
if (p_lock_switch == 1) {
    p_lock_increment++;
    p_lock_increment = p_lock_increment % p_lock_size; // Wrap around
}
🎬 Recording Process

When user presses "Record P-Lock" button:

  1. p_lock_switch is enabled
  2. All current parameter values are copied to ALL time steps (freeze frame)
  3. As user moves MIDI controllers, values are written to current time step
  4. p_lock_increment advances each frame
  5. After 240 steps, it loops back to step 0
  6. The recorded motion plays back indefinitely!
// When P-Lock recording is enabled (MIDI button press)
if (message.control == 55) { // P-Lock record button
    if (message.value == 127) { // Button pressed
        p_lock_switch = 1; // Enable recording

        // FREEZE FRAME: Copy current values to all steps
        // This gives us a baseline to record over
        for (int i = 0; i < p_lock_number; i++) {
            p_lock_smoothed[i] = 0; // Reset smoothed values

            for (int j = 0; j < p_lock_size; j++) {
                // Copy current value to every time step
                p_lock[i][j] = p_lock[i][p_lock_increment];
            }
        }
    }

    if (message.value == 0) { // Button released
        p_lock_switch = 0; // Stop recording, start playback
    }
}

Frame Processing Flow

Complete Rendering Pipeline

1. update()
   ├─ fft.update()                    (Analyze audio)
   ├─ cam1.update()                   (Grab camera frame)
   ├─ midibiz()                       (Process MIDI)
   └─ p_lockUpdate()                  (Update automation)

2. draw()
   ├─ fftAssignValues()               (Get FFT bands)
   ├─ parametersAssign()              (Calculate all parameters)
   │
   ├─ framebuffer0.begin()
   │  ├─ Draw camera input
   │  ├─ shader_mixer.begin()
   │  │  ├─ Set uniforms (parameters)
   │  │  ├─ Set textures (delayed frames)
   │  │  └─ GPU processes every pixel
   │  └─ shader_mixer.end()
   │  └─ framebuffer0.end()
   │
   ├─ sharpenFramebuffer.begin()
   │  ├─ Draw framebuffer0
   │  ├─ shaderSharpen.begin()
   │  │  └─ GPU sharpens image
   │  └─ shaderSharpen.end()
   │  └─ sharpenFramebuffer.end()
   │
   ├─ sharpenFramebuffer.draw()       (Display to screen)
   │
   ├─ pastFrames[N].begin()
   │  └─ Store current frame           (Save to circular buffer)
   │  └─ pastFrames[N].end()
   │
   └─ incIndex()                       (Advance buffer position)

The Update Function

// update() is called BEFORE draw() every frame
// This is where we update state, not render graphics

void ofApp::update() {
    /********** STEP 1: UPDATE AUDIO ANALYSIS **********/
    fft.update(); // Analyze latest audio buffer from sound card

    /********** STEP 2: UPDATE VIDEO INPUT **********/
    inputUpdate(); // Grab latest camera frame

    /********** STEP 3: PROCESS MIDI MESSAGES **********/
    midibiz(); // Handle all MIDI received since last frame

    /********** STEP 4: UPDATE AUTOMATION **********/

    // Used for auto-normalizing audio reactivity

The Draw Function (Simplified Walkthrough)

void ofApp::draw() {
    /********** PREPARATION PHASE **********/

    // STEP 1: Process audio to get smoothed frequency bands
    fftAssignValues();

    // STEP 2: Calculate all effect parameters
    // This combines MIDI, P-Locks, and audio reactivity
    parametersAssign();

    /********** PRIMARY EFFECTS PASS **********/

    // STEP 3: Begin rendering to framebuffer0
    framebuffer0.begin();

    // STEP 4: Activate the mixer shader
    shader_mixer.begin();

    // STEP 5: Draw camera input (this becomes tex0 in shader)
    cam1.draw(0, 0, 640, 480);

    // STEP 6: Send all parameters to shader
    // These are called "uniforms" - values that stay constant per frame
    shader_mixer.setUniformTexture("fb",
        pastFrames[d_delay].getTexture(), 1);
    shader_mixer.setUniformTexture("temporalFilter",
        pastFrames[(abs(framedelayoffset - framebufferLength) - 1) %
        framebufferLength].getTexture(), 2);

    // Send all continuous parameters
    shader_mixer.setUniform1f("lumakey", d_lumakey_value);
    shader_mixer.setUniform1f("fbMix", d_mix);
    shader_mixer.setUniform1f("fbHue", d_hue);
    shader_mixer.setUniform1f("fbSaturation", d_sat);
    shader_mixer.setUniform1f("fbBright", d_bright);
    // ... (many more parameters)

    // Send switches
    shader_mixer.setUniform1i("toroidSwitch", toroidSwitch);
    shader_mixer.setUniform1i("mirrorSwitch", mirrorSwitch);
    // ... (more switches)

    // STEP 7: Shader processes every pixel in parallel!
    shader_mixer.end();
    framebuffer0.end();

    /********** SHARPENING PASS **********/

    // STEP 8: Apply sharpening to processed image
    sharpenFramebuffer.begin();
    shaderSharpen.begin();
    framebuffer0.draw(0, 0); // Draw processed image
    shaderSharpen.setUniform1f("sharpenAmount", d_sharpenAmount);
    shaderSharpen.setUniform1f("vSharpenAmount", vSharpenAmount * c_sharpenAmount);
    shaderSharpen.end();
    sharpenFramebuffer.end();

    /********** DISPLAY TO SCREEN **********/

    // STEP 9: Draw final result to screen
    sharpenFramebuffer.draw(0, 0, ofGetWidth(), ofGetHeight());

    /********** SAVE TO CIRCULAR BUFFER **********/

    // STEP 10: Store this frame for future delay effects
    int writeIndex = abs(framebufferLength - framedelayoffset) - 1;
    pastFrames[writeIndex].begin();

    if (wet_dry_switch == 1) {
        // "Wet" mode: Store the processed output (feedback!)
        sharpenFramebuffer.draw(0, 0);
    } else {
        // "Dry" mode: Store raw camera input (traditional delay)
        cam1.draw(0, 0, 640, 480);
    }

    pastFrames[writeIndex].end();

    /********** ADVANCE TIME **********/

    // STEP 11: Move to next position in circular buffer
    incIndex();

    // STEP 12: Advance P-Lock position if recording/playing
    if (p_lock_switch == 1) {
        p_lock_increment++;
        p_lock_increment = p_lock_increment % p_lock_size;
    }
}
🎯 Understanding "Wet" vs "Dry" Feedback

Dry mode (traditional video delay):

  • Store clean camera frames in buffer
  • Mix clean past frames with current frame
  • Result: Classic "echo" effect

Wet mode (feedback loop):

  • Store processed output in buffer
  • Process past processed frames again
  • Result: Compounding, evolving, chaotic effects
  • Like pointing a camera at its own monitor!

Wet mode can create infinite tunnels, fractals, and psychedelic patterns because effects stack on themselves.

Header Files vs Implementation Files

📚 Why Separate Headers and Implementation?

Historical reasons:

  • Speeds up compilation (only recompile changed files)
  • Allows circular dependencies
  • Hides implementation details

Think of it like:

  • .h file = Book cover and table of contents
  • .cpp file = The actual chapters

Class Inheritance

// Our class inherits from TWO base classes
class ofApp : public ofBaseApp, public ofxMidiListener {

// What does this mean?
// 1. ofBaseApp provides setup(), update(), draw() structure
// 2. ofxMidiListener provides newMidiMessage() callback
// 3. ofApp MUST implement these methods

// This is called MULTIPLE INHERITANCE
⚠️ Virtual Functions & Polymorphism

When a class inherits from a base class with virtual functions, it can override them. openFrameworks calls these overridden functions automatically:

  • oF calls setup() → your ofApp::setup() runs
  • oF calls update() → your ofApp::update() runs
  • MIDI system calls newMidiMessage() → your ofApp::newMidiMessage() runs

This is the Template Method Pattern - the framework provides the structure, you fill in the behavior.

Arrays vs Vectors

// C-style array (fixed size, allocated at compile time)
ofFbo pastFrames[60];

// Vector (dynamic size, can grow/shrink)
std::vector<ofxMidiMessage> midiMessages;

// When to use each?
// ARRAY: Size known at compile time, need max performance
// VECTOR: Size varies, need flexibility

// Examples:

// Array access (fast, but no bounds checking!)
pastFrames[5].draw(0, 0);

// Vector access
midiMessages.push_back(newMessage);  // Add to end
midiMessages.erase(midiMessages.begin());  // Remove from start
int size = midiMessages.size();  // Get current size

Pass by Reference

// Pass by VALUE (copies the data)
void processMessage(ofxMidiMessage msg) {
    // 'msg' is a COPY - changes don't affect original
}

// Pass by REFERENCE (works with original data)
void processMessage(ofxMidiMessage& msg) {
    // 'msg' refers to original - changes DO affect it
    // MUCH faster for large objects (no copying!)
}

// Pass by CONST REFERENCE (read-only, no copy)
void processMessage(const ofxMidiMessage& msg) {
    // Best of both: fast + safe (can't modify original)
}
✅ Reference Best Practice

Rule of thumb:

  • Primitives (int, float, bool): Pass by value
  • Objects you modify: Pass by reference &
  • Objects you read: Pass by const reference const &

This code uses references extensively for MIDI messages to avoid copying large data structures every frame.

// #define creates a text substitution
